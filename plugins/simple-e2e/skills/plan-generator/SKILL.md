# Plan Generator Skill

prd.md와 sdd.md를 분석하여 TDD plan.md를 일관된 포맷으로 생성한다.

## 참고 자료

- 출력 템플릿: [template.md](template.md)
- 완성 예시: [examples/sample.md](examples/sample.md)

---

## 실행 절차

### Step 1: PRD 및 SDD 읽기

```
1. prd.md (또는 사용자가 지정한 PRD 파일)를 읽는다
2. sdd.md (시스템 설계서)를 읽는다
3. prd.md에서 아래 항목을 추출한다:
   - 프로젝트명
   - 코드 위치 (디렉토리 경로)
   - 기술 스택
   - 핵심 기능 목록
   - 기능별 요구사항 (입력, 출력, 제약조건, 예외)
   - 우선순위 또는 단계 구분 (있는 경우)
   - 사용자 스토리 (있는 경우)
4. sdd.md에서 아래 항목을 추출한다:
   - 전체 시스템 아키텍처 (컴포넌트 간 의존 관계)
   - 데이터베이스 모델링 (테이블/모델 목록)
   - API 인터페이스 명세 (엔드포인트 목록)
   - 폴더 구조 및 컴포넌트 분리 전략
   - 보안 설계 (인증/인가 흐름, 미들웨어)
   - 예외 처리 및 에러 전략
5. 기존 plan.md가 있으면 함께 읽어 중복 방지
```

### Step 2: 사용자 확인 — 누락 정보 질문

PRD와 SDD에서 다음 정보가 명시되지 않은 경우 사용자에게 질문한다. plan.md 생성에 필수적인 정보이므로 추정하지 않고 반드시 확인한다.

```
확인 항목:
- 프로젝트명 (코드 디렉토리명으로 사용됨)
- 코드 위치 (예: src/, pkg/, cart/)
- 기술 스택 (언어 + 테스트 프레임워크)
  예: "Go + go test", "TypeScript + Vitest", "Python + pytest"

질문 예시:
"plan.md 생성을 위해 몇 가지 확인이 필요합니다:
 1. 프로젝트명: (예: cart-calculator)
 2. 코드 위치: (예: src/)
 3. 기술 스택: (예: Go + go test)
"

PRD/SDD에 이미 명시된 항목은 다시 묻지 않는다.
```

### Step 3: Phase 분류

PRD의 기능 요구사항과 SDD의 아키텍처/컴포넌트 구조를 종합하여 Phase로 묶는다:

```
분류 기준:
- sdd.md의 시스템 아키텍처와 폴더 구조를 기반으로 레이어/모듈 단위로 Phase 구성
- sdd.md의 컴포넌트 간 의존 관계를 반영하여 의존되는 쪽을 먼저 배치
- sdd.md의 DB 모델링이 있으면 DB/스키마 관련 Phase를 비즈니스 로직보다 앞에 배치
- sdd.md의 API 명세를 참조하여 엔드포인트별로 Step을 구체화
- Phase 1은 항상 프로젝트 구조 및 기반 (초기화, 디렉토리, 기본 구조체/타입)

Phase 이름 규칙:
- Phase 1, Phase 2, Phase 3... 순서로 번호 부여
- 이름은 "기능 영역 (코드 위치)" 형태
  예: "상품 합계 계산 (cart/)"
  예: "할인 적용 (cart/)"
  예: "프로젝트 구조 및 기반"
```

### Step 4: Step 도출

각 Phase 안에서 TDD 사이클에 따라 Step을 도출한다:

```
Step 구성 규칙:
- Step 번호는 Phase 상관없이 전체 순번으로 매긴다 (Step 1, Step 2, ... Step N)
- 프로젝트 초기화, 설정 등 비-TDD Step도 포함 가능
- 테스트 케이스는 나열하지 않는다 — 기능 정의만 기술
  (구체적인 테스트는 tdd-cycle 실행 시 도출)

Step 기술 형식:
- [ ] Step N: {설명} — {세부 내용}

설명: 한 줄로 간결하게 (예: "상품 합계 구현", "입력 검증 구현")
세부 내용: 대상 함수명, 구조체명, 구현할 기능을 쉼표로 나열
  예: "CalculateSubtotal, 빈 장바구니/단일/복수 상품, 수량 곱셈"
  예: "가격 0 이하 에러, 수량 0 이하 에러"

주의:
- 개별 테스트 이름이나 테스트 시나리오는 포함하지 않는다
- 세부 내용은 구현할 것의 "무엇"을 기술하지, "어떻게 테스트할지"는 기술하지 않는다
```

### Step 5: 스토리 매핑

PRD의 기능 요구사항과 구현 Step을 매핑하는 테이블을 작성한다:

```
스토리 매핑 규칙:
- PRD의 기능 요구사항(또는 사용자 스토리)을 행으로 나열
- 해당 기능을 구현하는 Step 번호를 매핑
- 하나의 기능이 여러 Step에 걸칠 수 있음
- 매핑 테이블의 열 헤더: "기능 요구사항 | 구현 Step"
```

### Step 6: plan.md 생성

[template.md](template.md)의 포맷을 따라 plan.md를 작성한다:

```
필수 포함 항목:
✅ 제목: "# 코드 생성 계획 — {프로젝트명}"
✅ 프로젝트 정보: 프로젝트명, 코드 위치, 기술 스택, TDD 적용 여부
✅ 코드 생성 순서: Phase별 섹션 + Step 목록
✅ 스토리 매핑 테이블
✅ 모든 Step은 "- [ ]" 체크박스 형식

순서 원칙:
- 기반/인프라 → 도메인 → 비즈니스 로직 → UI/CLI → 빌드
- 의존성 없는 것 → 있는 것

저장 위치: 프로젝트 루트에 plan.md로 저장
```

### Step 7: 검토 및 확인

생성 후 사용자에게 요약 보고:

```
📋 plan.md 생성 완료

Phase: {N}개
총 Step: {N}개

Phase 1 {Phase명}: Step 1-{N}
Phase 2 {Phase명}: Step {N+1}-{M}
...

수정하거나 추가할 내용이 있으면 말씀해주세요.
준비가 되면 "go"를 입력해 첫 번째 Step을 시작할 수 있습니다.
```

---

## 기존 plan.md 업데이트

prd.md 또는 sdd.md가 변경된 경우 plan.md를 업데이트한다:

```
규칙:
- [x] 완료된 항목은 절대 수정하지 않는다
- [ ] 미완료 항목은 PRD 변경에 맞게 수정 가능
- 새 요구사항은 적절한 Phase에 Step 추가
- 삭제된 요구사항의 Step은 취소선(~~텍스트~~)으로 표시 후 사유 주석 추가
```

## 예외 상황

| 상황 | 대응 |
|---|---|
| prd.md가 없음 | "prd.md가 없습니다. 먼저 PRD를 작성해주세요. 'PRD 만들어줘'를 입력하세요." |
| sdd.md가 없음 | "sdd.md가 없습니다. 먼저 설계서를 작성해주세요. '설계서 만들어줘'를 입력하세요." |
| prd.md와 sdd.md 간 불일치 | 사용자에게 어느 쪽을 기준으로 할지 확인 |

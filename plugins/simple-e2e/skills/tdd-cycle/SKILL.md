# TDD Cycle Skill

## "go" 명령 실행 절차

### Step 0: 개발 단계 확인 (사전 점검)

```
1. sdd.md와 prd.md가 존재하는지 확인한다
2. 아래 4단계 개발 순서를 참조한다 (한 단계가 완료되기 전에 다음 단계로 넘어가지 않는다):

   [1단계: 프로젝트 환경 초기화 및 구조 세팅]
   - 의존성 패키지 설치 (package.json, 설치 명령어)
   - sdd.md에 정의된 폴더 구조 생성
   - 기본 설정 파일 작성 (tsconfig, eslint, prettier, tailwind, next.config 등)
   - .env.example 생성 (실제 시크릿은 절대 하드코딩 금지)
   - Docker 관련 파일 (docker-compose.yml 등)
   - 이 단계의 테스트: 빌드/린트가 에러 없이 통과하는지 확인

   [2단계: 데이터베이스 연동 및 공통 인터페이스/유틸리티 구현]
   - Prisma 스키마 작성 (sdd.md의 DB 모델링 기반)
   - 마이그레이션 실행
   - 공통 타입/인터페이스 정의
   - 공통 유틸리티 (crypto, sanitizer, validator 등)
   - 에러 처리 미들웨어 (RFC 7807)
   - 인증/인가 미들웨어
   - 이 단계의 테스트: DB 연결, 마이그레이션, 유틸리티 단위 테스트

   [3단계: 핵심 비즈니스 로직 및 백엔드 API 구현]
   - Service 레이어 (비즈니스 로직)
   - Controller 레이어 (요청/응답 처리)
   - Route 정의
   - 위험 키워드 감지, 알림, 리포트 등 부가 서비스
   - 이 단계의 테스트: API 엔드포인트별 단위/통합 테스트

   [4단계: 프론트엔드 주요 화면(UI) 컴포넌트 및 API 연동]
   - 페이지 컴포넌트 (App Router)
   - 재사용 UI 컴포넌트
   - Zustand 스토어
   - API 서비스 레이어 (프론트)
   - 커스텀 훅
   - 이 단계의 테스트: 컴포넌트 렌더링 테스트, 통합 테스트

3. plan.md를 읽고, 현재 진행 중인 단계를 파악한다
   - plan.md의 섹션 헤더(## 1단계, ## 2단계 등)로 단계를 구분한다
   - 현재 단계의 모든 [ ] 항목이 [x]로 변경되었으면 → 단계 완료
4. 단계 완료 시:
   - 실행/테스트 방법을 안내한다
   - "다음 단계 진행해"라는 사용자 지시를 기다린다
   - 사용자 지시 없이 다음 단계로 넘어가지 않는다
5. sdd.md에 정의된 아키텍처, DB 스키마, API 명세, 폴더 구조를 참조하여
   테스트와 구현 코드가 설계서와 일치하도록 한다
```

### 단계별 적용 규칙

```
- plan.md의 테스트 항목은 위 4단계 순서에 맞게 배치되어야 한다
- tdd-cycle에서 "go" 실행 시, 현재 단계에 해당하는 테스트만 진행한다
- 각 단계가 끝나면 실행/테스트 방법을 안내하고 사용자 확인을 기다린다
- 사용자가 "다음 단계 진행해"라고 지시해야 다음 단계로 이동한다
- 코드 작성 시 최신 모범 사례(Best Practices) 적용, 재사용 가능한 모듈 분리
- 주석은 한국어로 친절하게 작성
- 중복 코드 방지를 위해 공통 모듈을 적극 활용
```

### Step 1: plan.md에서 다음 테스트 찾기

```
1. plan.md를 읽는다
2. 현재 단계 내의 [ ] 항목 중 첫 번째를 선택한다
   (다음 단계의 항목은 선택하지 않는다)
3. 선택한 항목을 사용자에게 확인한다:
   "📍 현재 단계: N단계 — [단계명]"
   "다음 테스트: [항목 내용] — 진행할까요?"
4. 확인 후 진행
```

### Step 2: RED — 실패하는 테스트 작성

```
1. 테스트 파일에 새 테스트 함수 추가
2. 테스트 이름: should[ExpectedBehavior]When[Condition]
   예: shouldReturnZeroWhenListIsEmpty, shouldThrowExceptionWhenInputIsNegative
3. 한 번에 하나의 assert만 작성 (가능한 경우)
4. 구현 코드는 작성하지 않는다
5. 테스트 실행 → 실패 메시지 출력 (컴파일 에러는 "Red"로 인정하지 않음 — 테스트가 실행되어 실패해야 함)
6. "❌ RED: [실패 내용]" 형식으로 보고
```

**출력 예시:**
```
❌ RED
테스트: shouldReturnSumWhenGivenTwoNumbers
실패 이유: TypeError: add is not a function
```

### Step 3: GREEN — 최소 구현

```
1. 테스트를 통과시키는 **가장 단순한** 코드 작성
2. "너무 단순하지 않나?" 싶을 정도가 적당하다
3. 하드코딩도 허용 — 이후 테스트가 일반화를 강제할 것이다
4. 프로덕션 코드의 우아함은 이 단계의 목표가 아니다
5. 테스트 실행 → 통과 확인
6. "✅ GREEN: 모든 테스트 통과" 형식으로 보고
7. 테스트 통과 실패 시 Step 3 반복 (구현 수정)
```

**출력 예시:**
```
✅ GREEN
구현: add(a, b) { return a + b }
통과: 3/3 tests passing
```

### Step 4: REFACTOR — 구조 개선

```
중복이나 불명확한 이름이 있는 경우에만 수행:
1. 반드시 모든 테스트가 통과한 상태에서만 시작
2. 한 번에 하나의 리팩토링만 수행
3. 적용할 리팩토링 패턴 이름 명시 (예: "Extract Method", "Rename Variable", "Inline Temp")
4. 변경 사항 설명
5. 각 리팩토링 후 즉시 테스트 재실행 → 여전히 통과 확인
6. "🔧 REFACTOR: [패턴명]" 형식으로 보고

필요 없으면 이 단계 건너뜀
```

**출력 예시:**
```
🔧 REFACTOR: Extract Method
변경: 검증 로직을 validateInput()으로 추출
결과: 3/3 tests still passing
```

### Step 5: 커밋

```
테스트 + 구현이 함께인 경우:
  git add -A
  git commit -m "[behavioral] should[ExpectedBehavior]When[Condition]"

리팩토링이 있는 경우 별도 커밋:
  git add -A
  git commit -m "[structural] Extract Method: validateInput"
```

### Step 6: plan.md 업데이트

```
1. 완료된 항목의 [ ]를 [x]로 변경
2. 완료 보고:

📋 진행 현황
완료: [x] 항목명
남은 테스트: N개
다음: "go"를 입력하면 다음 테스트를 진행합니다.
```

## 흐름 요약

```
plan.md 읽기
    ↓
[ ] 항목 선택 → 사용자 확인
    ↓
❌ RED: 실패 테스트 작성 & 실행
    ↓
✅ GREEN: 최소 구현 & 테스트 통과
    ↓
🔧 REFACTOR: 필요 시 구조 개선 (선택)
    ↓
💾 COMMIT: [behavioral] + 필요 시 [structural] 별도
    ↓
📋 plan.md [x] 마킹 & 현황 보고
```

## 단계 전환 규칙

```
현재 단계의 모든 테스트가 [x]로 완료되면:

1. 단계 완료 보고:
   "🏁 N단계 완료: [단계명]"
   "완료된 테스트: M개"

2. 실행/테스트 방법 안내:
   - 1단계 완료 시: "npm run build && npm run lint 로 확인해주세요"
   - 2단계 완료 시: "npx prisma migrate dev && npm test 로 확인해주세요"
   - 3단계 완료 시: "npm test && curl로 API 엔드포인트 테스트해주세요"
   - 4단계 완료 시: "npm run dev 로 전체 동작 확인해주세요"

3. 사용자 지시 대기:
   "확인 후 '다음 단계 진행해'를 입력해주세요."

4. 사용자가 "다음 단계 진행해"라고 하기 전까지 다음 단계 테스트 진행 금지
```

## 설계서 참조 규칙

```
코드를 작성할 때 반드시 sdd.md를 참조한다:
- DB 모델링 → Prisma 스키마, 타입 정의
- API 명세 → 라우트, 컨트롤러, 요청/응답 타입
- 폴더 구조 → 파일 생성 위치
- 보안 설계 → 미들웨어, 검증 로직
- 에러 전략 → RFC 7807 에러 응답 포맷

코드 품질:
- 최신 모범 사례(Best Practices) 적용
- 재사용 가능한 모듈로 분리하여 중복 방지
- 주석은 한국어로 친절하게 작성
```

## 절대 금지 사항

```
- 테스트 없이 프로덕션 코드 작성 금지
- Structural 변경과 Behavioral 변경을 같은 커밋에 혼합 금지
- 실패하는 테스트가 있는 상태에서 리팩토링 금지
- 테스트를 통과시키기 위해 테스트 자체를 수정 금지
```

## 커밋 전 체크리스트

```
1. 모든 테스트 통과 확인 (long-running 테스트 제외)
2. 컴파일러/린터 경고 0개 확인
3. 커밋 메시지에 반드시 [structural] 또는 [behavioral] 접두사 포함
```

## Tidy First 원칙

```
- STRUCTURAL: 이름 변경, 메서드 추출, 코드 이동 — 동작 변경 없음
- BEHAVIORAL: 기능 추가 또는 수정 — 반드시 테스트 먼저
- 둘 다 필요한 경우: Structural 먼저, 별도 커밋 후 Behavioral 진행
```

## 코드 품질

```
- 중복을 발견하면 즉시 제거 (단, Green 상태일 때만)
- 메서드는 하나의 책임만 가진다
- 이름은 의도를 명확히 드러내야 한다
```

## 결함 수정 규칙

```
결함 발견 시 순서:
1. API 레벨 실패 테스트 먼저 작성
2. 결함을 재현하는 최소 단위 테스트 작성
3. 두 테스트를 모두 통과시키는 코드 구현
```

## 테스트 범위 규칙

```
- 단위 테스트: 매 사이클마다 실행
- 통합 테스트: Green 달성 후 실행
- Long-running 테스트: 커밋 전에만 실행
- 테스트 커버리지: 새로운 동작은 반드시 테스트로 커버
```

## TDD 필수 규칙 (NON-NEGOTIABLE)

```
- 프로덕션 코드가 변경되는 어떤 요청이든 TDD 사이클을 절대 생략하지 않는다
- 프로덕션 코드 파일(.go, .ts, .tsx)을 생성하면 반드시 같은 작업 내에서 테스트 파일도 함께 생성한다
- agent에게 코드 생성을 위임할 때 프롬프트에 "테스트도 함께 작성하라"는 지시를 반드시 포함한다
- 작업 완료 전 `make test` 또는 `npx vitest run`으로 전체 테스트 통과를 확인한다
```

## 테스트 커버리지 규칙

```
- 모든 프로덕션 코드의 테스트 커버리지는 100%를 목표로 작성한다
- 새로운 코드를 작성하면 해당 코드에 대한 테스트를 반드시 함께 작성한다
- 기존 코드를 수정하면 커버리지가 떨어지지 않도록 테스트도 함께 수정/추가한다
- Go 프로젝트: `go test ./... -cover`로 커버리지를 확인한다
- WebUI 프로젝트: `npx vitest run --coverage`로 커버리지를 확인한다
- 커버리지 리포트에서 Uncovered Line이 발견되면 해당 라인을 커버하는 테스트를 추가한다
- API 호출이 필요한 컴포넌트는 모킹(vi.mock)을 사용하여 테스트한다
```

## 예외 상황 처리

| 상황 | 대응 |
|---|---|
| plan.md가 없음 | "plan.md가 없습니다. 생성해드릴까요?" |
| sdd.md가 없음 | "sdd.md가 없습니다. 먼저 설계서를 작성해주세요." |
| 모든 테스트 완료 | "🎉 plan.md의 모든 테스트가 완료되었습니다." |
| 현재 단계 테스트 모두 완료 | 단계 전환 규칙에 따라 사용자 지시 대기 |
| 테스트가 예상과 다르게 실패 | 사용자에게 보고 후 방향 확인 |
| 구현 방향이 불명확 | "이 테스트의 의도를 확인해주세요: [내용]" |
| 설계서와 plan.md가 불일치 | 사용자에게 어느 쪽을 따를지 확인 |
